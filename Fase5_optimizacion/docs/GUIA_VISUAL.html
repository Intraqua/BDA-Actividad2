Detalle de Capturas Clave

CAPTURA 02, 03, 04 - EXPLAIN ANALYZE (ANTES)

Estas capturas son criticas para demostrar el problema:
  1. Buscar Seq Scan - indica escaneo completo de tabla
  2. Buscar Hash Join - operacion costosa en memoria
  3. Anotar actual time=XXX - tiempo real de ejecucion
  4. Verificar rows= - filas procesadas vs estimadas
CAPTURA 08, 09 - EXPLAIN ANALYZE (DESPUES)

Estas capturas demuestran la mejora:
  1. Buscar Index Scan o Index Only Scan
  2. Verificar reduccion en actual time
  3. Confirmar uso de vista materializada en el plan
  4. Comparar Buffers antes vs despues
Importante: Los tiempos reales dependen del volumen de datos y hardware. Lo importante es demostrar la mejora relativa (porcentaje de reduccion), no los valores absolutos.

Comandos SQL Esenciales

Verificar plan de ejecucion

EXPLAIN (ANALYZE, BUFFERS, FORMAT TEXT)
SELECT ... FROM ... WHERE ...;

Ver indices existentes

SELECT indexname, tablename, indexdef 
FROM pg_indexes 
WHERE schemaname = 'public';

Verificar vistas materializadas

SELECT matviewname, ispopulated 
FROM pg_matviews 
WHERE schemaname = 'public';

Estadisticas de uso de indices

SELECT indexrelname, idx_scan, idx_tup_read
FROM pg_stat_user_indexes
WHERE schemaname = 'public'
ORDER BY idx_scan DESC;

Refrescar vista materializada

-- Refresco normal (bloquea lecturas)
REFRESH MATERIALIZED VIEW mv_ventas_categoria_mes;

-- Refresco concurrente (requiere UNIQUE INDEX)
REFRESH MATERIALIZED VIEW CONCURRENTLY mv_ventas_categoria_mes;

Interpretacion de EXPLAIN ANALYZE

Indicadores de problema:
  • Seq Scan en tablas grandes = falta indice
  • Sort con Sort Method: external merge = ordenacion en disco
  • Hash Join con Batches > 1 = memoria insuficiente
  • rows=X (actual rows=Y) con X muy diferente de Y = estadisticas desactualizadas
Indicadores de optimizacion exitosa:
  • Index Scan o Index Only Scan
  • Bitmap Index Scan para multiples condiciones
  • Reduccion significativa en actual time
  • Reduccion en Buffers: shared hit/read
